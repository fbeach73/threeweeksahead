0Overview

Adding any new site to SES is the same pattern. Production access is account-wide and already granted — never need to wait again. For each new site:

  1. Verify the domain in SES (DKIM + custom MAIL FROM).
  2. Add DNS records in Cloudflare (3 DKIM CNAMEs + bounce MX/SPF + DMARC).
  3. Reuse the IAM keys from rpl-ses-sender — same access works for every domain in this account.
  4. Wire the code — either copy the RPL pattern for transactional, or for a one-pager just a single Vercel function.
  5. Set Vercel env vars and deploy.
  6. Smoke test — send + verify.

For higher-traffic sites (or where you want delivery/bounce/complaint logging to your DB), also do step 7.

1Verify domain in SES

  1. Open SES → Verified identities (region US East — N. Virginia top-right).
  2. Click Create identity → Domain.
  3. Enter {{domain}}.
  4. Under Advanced DKIM settings, Identity type: pick Easy DKIM (not Deterministic Easy DKIM, which is for cross-region replication, nor BYODKIM, which is Mailgun-style bring-your-own-key).
  5. DKIM signing key length: pick RSA_2048_BIT (AWS-recommended; no downside — doesn't affect the CNAMEs).
  6. Publish DNS records to Route53: ignore — only relevant if DNS were on Route 53. Yours is Cloudflare, so this has no effect.
  7. DKIM signatures: keep Enabled.
  8. Tick Use a custom MAIL FROM domain. MAIL FROM subdomain: bounce.{{domain}}. Leave Behavior on MX failure as default.
  9. For DMARC the form pre-fills the TXT — fine, leave p=none; as the value.
  10. Click Create identity. SES gives you a screen of DNS records — keep that tab open for step 2.
Reminder: SES uses CNAME for DKIM (different from Mailgun's TXT). Don't go looking for a public key/hash — Amazon holds it.

2Cloudflare DNS records

Open Cloudflare → {{domain}} → DNS → Records. Add these (copy each row from the SES identity page):

#TypeNameValueProxy
1–3CNAMExxxx._domainkey (3 of them)xxxx.dkim.amazonses.comDNS only
4MXbouncefeedback-smtp.us-east-1.amazonses.com · priority 10DNS only
5TXTbouncev=spf1 include:amazonses.com ~allDNS only
6TXT_dmarcv=DMARC1; p=none;DNS only
Two Cloudflare gotchas that will burn you:
  • Name field: type just xxxx._domainkey, not xxxx._domainkey.{{domain}}. Cloudflare auto-appends the zone. Double-appending makes the record invisible.
  • DKIM CNAMEs must be grey-clouded (DNS only). Orange proxied = DKIM breaks = no verification.

After saving, refresh the SES identity page. Identity status should flip to Verified and DKIM to Successful within a few minutes on Cloudflare.

3IAM credentials — reuse rpl-ses-sender

Don't create a new IAM user per site. The existing rpl-ses-sender has account-wide SES send permission, which covers every verified domain in the account.

You'll reuse these two values across every site:

AWS_ACCESS_KEY_ID       = (rpl-ses-sender key)
AWS_SECRET_ACCESS_KEY   = (rpl-ses-sender secret)
AWS_REGION              = us-east-1

If you lost the secret, you can't retrieve it — delete the access key in IAM and create a new one.

4Code integration

Option A — Single Vercel function (simple sites / one-pagers)

Static HTML + one serverless function at api/subscribe.js (or similar). The threeweeksahead repo is the canonical example. Send via @aws-sdk/client-sesv2, store subscribers in Neon.

Don't forget the table. The Neon SQL Editor needs the schema run once — without it the function returns "Could not save subscription". Paste this: 
CREATE TABLE IF NOT EXISTS subscribers (
  id BIGSERIAL PRIMARY KEY,
  email TEXT NOT NULL UNIQUE,
  source TEXT,
  status TEXT NOT NULL DEFAULT 'active',
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  unsubscribed_at TIMESTAMPTZ
);
CREATE INDEX IF NOT EXISTS subscribers_status_idx ON subscribers(status);

Option B — Provider switch + webhook (apps with multiple email types)

RPL is the canonical example. Pattern:

  • src/lib/ses.ts — mirrors the Mailgun send signature; uses Raw MIME for PDF attachments.
  • src/lib/email-provider.ts — switches on EMAIL_PROVIDER=ses|mailgun. Defaults to mailgun for instant rollback.
  • One aliased import in email.ts; mailgun.ts stays untouched.
  • Drop in /api/webhooks/ses and the email_events Drizzle table only if doing step 7.

For newsletter sends

Don't build per-site. Use the central hub (fork of newsletter-pro) once it's wired to SES. Each site still sends from its own news.{{domain}} subdomain for reputation separation — the hub authenticates with SES and picks the right From/config set.

5Vercel env vars

In Vercel → the project → Settings → Environment Variables (Production scope; also Preview if you want):

Minimal (simple transactional, no event tracking)

EMAIL_PROVIDER          = ses          # only needed if using the provider switch
AWS_REGION              = us-east-1
AWS_ACCESS_KEY_ID       = (rpl-ses-sender)
AWS_SECRET_ACCESS_KEY   = (rpl-ses-sender)
SES_FROM_EMAIL          = kyle@{{domain}}

With Neon storage (e.g. signup forms)

POSTGRES_URL            = (Neon connection string)

With event tracking (step 7)

SES_CONFIGURATION_SET   = <config-set-name>
SES_SNS_TOPIC_ARN       = arn:aws:sns:us-east-1:...:<topic-name>
After adding env vars, you must redeploy — Vercel doesn't apply new env to existing deployments. Either push a commit, or Deployments → ⋯ → Redeploy.

6Smoke test

If site has a test endpoint (e.g. RPL)

curl -s "https://www.{{domain}}/api/test-email?email=kyle@{{domain}}"

Expect {"success":true,...} and the email in your inbox.

If site has a signup form

curl -s -X POST "https://{{domain}}/api/subscribe" \
  -H "Content-Type: application/json" \
  -d '{"email":"kyle@{{domain}}"}'

Confirm event landed in DB (if step 7 enabled)

RPL has scripts/check_events.mjs — copy-paste it into any site's repo and adjust the table name. Or query Neon directly:

SELECT id, event_type, recipient, message_id, created_at
FROM email_events
ORDER BY id DESC LIMIT 10;

7(Optional) Event tracking — Config Set + SNS

Only worth doing for sites where you want delivery/bounce/complaint events streamed into your own database. Skip for one-pagers with a single welcome email; the SES dashboard's account-wide bounce/complaint rates are enough at low volume.

  1. Create SNS topic — SNS → Topics → Create → Standard → name e.g. <site>-ses-events. Copy the ARN.
  2. Subscribe the topic to your webhook: Protocol HTTPS, Endpoint https://www.{{domain}}/api/webhooks/ses (use www. if your apex 301/307-redirects). Leave raw delivery OFF. Your handler auto-confirms.
  3. Create Config Set in SES → name e.g. <site>-transactional.
  4. On the Config Set → Event destinations → Add. Events: Sends, Deliveries, Hard bounces, Complaints. Destination: Amazon SNS → pick your topic.
  5. In your app's code, ensure ConfigurationSetName is attached to every SendEmailCommand. (RPL's ses.ts reads from SES_CONFIGURATION_SET.)
  6. Set Vercel env: SES_CONFIGURATION_SET + SES_SNS_TOPIC_ARN → redeploy.

!Common gotchas

  • Cloudflare orange-cloud on CNAME → DKIM breaks. Always grey-cloud DKIM/MX/SPF/DMARC.
  • Double-append on Cloudflare name field → record never resolves. Type _dmarc, not _dmarc.{{domain}}.
  • US vs EU SES dashboard → completely separate. Confirm region us-east-1 in the top-right before doing anything.
  • Apex redirects (e.g. {{domain}} → www.{{domain}}) confuse SNS HTTPS subscriptions. Always subscribe to the www (or whatever's the 200, not the 307).
  • Sandbox vs production → granted once per AWS account, account-wide. New domains in the same account skip the wait.
  • "More information needed" on the SES home page is your production-access support case status, not a missing setup step. It changes when AWS reviews your reply.
  • Vercel env changes need a redeploy — they don't apply to existing deployments.
  • MAIL FROM subdomain conflict — don't pick a subdomain that already hosts another mail system (e.g. Mailgun on mail.{{domain}}). Use bounce..
  • VDM (Virtual Deliverability Manager) is on by default in newer SES accounts and adds a small per-message surcharge for open/click tracking. Disable in SES → VDM if you want pure $0.10/1k pricing.
